Microsoft Defender for Endpoint: Assessment Notes & Key Concepts

JUN 2025 ~8 MIN READ BRITTANY

After completing the Microsoft Defender for Endpoint (MDE) course and working through the final assessment, I wanted to document the key concepts — particularly the ones that were more nuanced than they first appeared.

// TL;DR Hardware-based isolation (Windows Defender Application Guard) is a browser-level protection. Automation levels control how much MDE auto-remediates. Always onboard devices through the Security Center, not Intune directly.

Hardware-Based Isolation

This tripped me up at first. Hardware-based isolation in MDE refers specifically to Windows Defender Application Guard (WDAG), which opens untrusted browser sessions in a lightweight Hyper-V container — isolating them from the host OS at the hardware virtualization level.

Key things to know:

  • It's browser-level isolation (Microsoft Edge), not full VM isolation
  • Requires a CPU with virtualization extensions (Intel VT-x / AMD-V)
  • Untrusted sites run in the container; files downloaded there are sandboxed
  • Application Guard is separate from Windows Sandbox (which is a full isolated environment)

Automation Levels

MDE's automated investigation and remediation (AIR) has three automation levels that control how much the system acts on its own:

  • No automated response — Alerts only; security team handles everything manually
  • Semi (require approval) — MDE investigates and recommends, but a human approves every remediation action
  • Full automation — MDE investigates and remediates without requiring human approval
// GOTCHA "Semi - require approval for core folders remediation" means only core folder actions need approval, not all remediations. Easy to misread this on an assessment.

Device Onboarding

Devices get onboarded to MDE through the Microsoft Defender Security Center (security.microsoft.com), not directly through Intune. Intune can be used to deploy the onboarding package, but MDE itself manages the enrolled devices.

Onboarding methods by device type:

  • Windows 10/11: Group Policy, Intune/MDM, local script, SCCM
  • macOS: Intune, JAMF, manual package
  • Linux: manual script or Ansible
  • Mobile (iOS/Android): Microsoft Defender for Endpoint app via Intune

MDE + Defender for Office 365 Integration

When MDE and Microsoft Defender for Office 365 (MDO) are both active, they share signals through Microsoft 365 Defender (now Microsoft Defender XDR). This means:

  • Email-based threats detected by MDO can trigger endpoint investigations in MDE
  • Unified incident queue — one alert can contain signals from both products
  • Automated cross-product investigation: if a phishing email is opened and malware executed, the investigation spans email + endpoint automatically

What I'd Review Again

The areas I'd spend more time on before a real certification exam:

  • Attack Surface Reduction (ASR) rules — there are a lot of them and their exact scopes matter
  • Indicator of Compromise (IoC) management and custom detections
  • The difference between blocking and auditing modes for ASR
  • Roles and RBAC within the Security Center
